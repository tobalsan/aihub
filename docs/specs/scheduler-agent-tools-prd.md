# Scheduler Agent Tools PRD

## Problem Statement

Agents can only manage scheduler cron jobs indirectly today: humans edit `cron/jobs.json` or use the scheduler CLI/API. That blocks agents from scheduling their own future work, updating their recurring jobs, or inspecting latest scheduled output without human help.

Users want any agent to create and manage its own cron jobs through custom agent tools, but only when the scheduler extension is enabled. Agents must not manage other agents' jobs.

There is also a related operational problem: manual cron file edits and config changes currently require a gateway restart before the running process sees them. Once agents can create and manage jobs, humans may still edit `cron/jobs.json` directly on the server, and those edits should become active without a full deploy/restart.

## Solution

Add scheduler-provided custom agent tools. When the scheduler extension is enabled, each running agent receives tools for listing, creating, updating, deleting, and reading latest output for its own scheduler jobs.

Tools operate on the caller agent only. Job ids are generated by the scheduler service. Create defaults to enabled jobs. Tools expose raw cron scheduling fields and optional session id override. They do not expose model overrides. Expected validation failures return structured `{ ok: false, error }` responses instead of tool failures.

Add gateway hot reload support for config and agent cron files. Config reload should refresh the loaded config, extension enabled state, extension tool injection, agent registry, and scheduler lifecycle without requiring process restart. Cron file reload should refresh scheduler in-memory state when an agent's `cron/jobs.json` changes outside the scheduler API/tools.

## User Stories

1. As an agent, I want to list my scheduled jobs, so that I can understand my existing recurring work.
2. As an agent, I want to create a cron job for myself, so that I can schedule future recurring work without human intervention.
3. As an agent, I want created jobs to be enabled by default, so that scheduled work starts firing after creation.
4. As an agent, I want the scheduler to generate job ids, so that I do not have to invent stable ids.
5. As an agent, I want to name a job, so that humans and future agent runs can recognize its purpose.
6. As an agent, I want to set a cron expression, so that I can define recurring cadence precisely.
7. As an agent, I want to set an IANA timezone, so that schedules run in the intended local time.
8. As an agent, I want to optionally set a schedule start time, so that first run timing can be anchored.
9. As an agent, I want to set the scheduled prompt message, so that future runs know what task to perform.
10. As an agent, I want to optionally set a session id, so that scheduled runs can continue a known session when appropriate.
11. As an agent, I want scheduled runs to use scheduler-generated unique sessions by default, so that recurring runs do not accidentally share state.
12. As an agent, I want to update a job name, so that I can clarify job purpose over time.
13. As an agent, I want to update a job enabled state, so that I can pause or resume scheduled work.
14. As an agent, I want to update a job schedule, so that I can change cadence without deleting and recreating the job.
15. As an agent, I want to update a job message, so that future scheduled runs perform revised work.
16. As an agent, I want to update a job session id override, so that I can move future runs into or out of a stable session.
17. As an agent, I want to delete a job by exact id, so that I can remove work that is no longer needed.
18. As an agent, I want to read the latest output from one of my jobs, so that I can inspect what happened in the most recent scheduled run.
19. As an agent, I want latest output reads to be bounded, so that tool results do not flood my context.
20. As an agent, I want validation errors returned as structured results, so that I can recover and try again.
21. As a user, I want scheduler tools hidden when the scheduler extension is disabled, so that disabled scheduler behavior is clear.
22. As a user, I want agents restricted to their own jobs, so that one agent cannot modify another agent's schedule.
23. As a user, I want agents to manage cron jobs through existing scheduler logic, so that CLI/API/tool behavior stays consistent.
24. As a user, I want job files to remain compatible with current manual editing and CLI workflows, so that no migration is needed.
25. As a user, I want tool-created jobs to persist in the same per-agent cron job files, so that all scheduler entrypoints see the same state.
26. As a user, I want tools to avoid model override support, so that model choice remains a human/API/CLI concern for now.
27. As a developer, I want tool handlers to reuse scheduler service methods, so that there is no parallel storage implementation.
28. As a developer, I want tool schemas to mirror existing scheduler request schemas where possible, so that validation remains coherent.
29. As a developer, I want tests around tool behavior, so that agent-visible scheduler behavior is stable.
30. As an operator, I want scheduler extension enabled state to control tool injection, so that rollout follows existing extension configuration.
31. As an operator, I want gateway config changes to hot reload, so that enabling or disabling scheduler tools does not require a restart.
32. As an operator, I want agent config changes to hot reload, so that newly added or changed agents are reflected in extension context and scheduler state.
33. As an operator, I want manual `cron/jobs.json` edits to hot reload, so that direct server edits become active without deploy or restart.
34. As an operator, I want scheduler timers to refresh after cron file changes, so that changed jobs fire at the new expected time.
35. As an operator, I want invalid hot-reloaded config or job files to fail safely, so that the last good running state is not destroyed by a bad edit.
36. As an agent, I want scheduler tools to appear or disappear after scheduler enablement changes, so that my available tools match current configuration.

## Implementation Decisions

- Add scheduler custom agent tools through the existing extension tool interface.
- Inject scheduler tools only when scheduler extension config is enabled.
- Tools are self-only: they infer the caller agent from execution context and expose no agent id argument.
- Provide narrow tools instead of one action multiplexer:
  - list jobs
  - create job
  - update job
  - delete job
  - get latest output
- Create uses scheduler-generated job ids.
- Create defaults to enabled jobs, matching existing scheduler behavior.
- Update supports all non-id user-editable fields: name, enabled, schedule, message, and session id.
- Delete requires exact job id only.
- Scheduling input uses raw cron expression, timezone, and optional start time.
- Payload input supports message and optional session id.
- When session id is omitted, scheduled runs continue using scheduler-generated unique sessions.
- Tools do not expose provider/model override fields.
- Latest output returns a bounded preview with caller-controlled maximum under a hard cap.
- Expected validation and not-found failures return structured `{ ok: false, error }` responses.
- Unexpected internal failures may still fail the tool call.
- Tool handlers should reuse scheduler service/store behavior rather than writing job files directly.
- Tool output should be concise and stable for agent consumption.
- Add gateway hot reload for main config and agent config inputs that currently bind at startup.
- Config hot reload should refresh cached config, rebuilt extension runtime state, extension contexts, enabled/disabled extension behavior, and agent registry data.
- Scheduler hot reload should refresh per-agent cron jobs from disk and recompute next run state.
- Cron file hot reload should use polling based on file mtimes, not filesystem watch events.
- Config hot reload should also use polling based on file mtimes.
- Poll interval is fixed at 5 seconds for MVP.
- Polling is preferred for server deploys because it is less fragile across atomic writes, symlink swaps, and filesystem watcher edge cases.
- Hot reload should prefer last-known-good behavior: invalid changed config or malformed cron files should surface warnings/errors and avoid replacing valid in-memory state with broken state.
- Scheduler extension enablement changes should update agent tool injection without requiring gateway restart.
- Scheduler service should expose a refresh/reload boundary that can be tested independently from file watching.

## Testing Decisions

Good tests should assert external behavior visible to agents and users, not implementation internals. Tests should call extension tool discovery/execution or the public tool handler surface, then assert returned tool definitions, persistence effects, and structured results.

Test these modules/behaviors:

- Scheduler extension tool discovery:
  - tools appear when scheduler is enabled
  - tools do not appear when scheduler is disabled
- Scheduler tool CRUD behavior:
  - create persists a self-owned job
  - list returns only caller agent jobs
  - update modifies allowed fields
  - delete removes exact job id
- Scheduler tool validation:
  - invalid cron/timezone/start time returns `{ ok: false, error }`
  - missing job id returns `{ ok: false, error }`
- Session behavior:
  - omitted session id preserves default scheduler-generated session behavior
  - provided session id is persisted in payload
- Output behavior:
  - latest output returns bounded preview
  - missing output returns structured not-found result
- Injection path compatibility:
  - existing Pi/container extension tool injection tests can be extended only if necessary; the extension runtime tests already cover generic injection.
- Gateway hot reload behavior:
  - config changes refresh loaded config without process restart
  - scheduler enabled/disabled changes update extension tool discovery
  - agent additions/removals refresh scheduler-visible agent set
  - malformed config changes keep last-known-good runtime state
- Scheduler cron reload behavior:
  - polling detects manual `cron/jobs.json` edits and refreshes in-memory job list
  - changed schedules recompute next run state
  - malformed job files do not destroy last-known-good jobs
  - new agent cron files become visible after agent/config reload

Prior art exists in other extension agent tool tests and extension runtime tests for custom tool discovery/execution, plus scheduler service/store tests for persistence and validation behavior.

## Out of Scope

- Cross-agent scheduler management.
- UI for agent-created scheduler jobs.
- Friendly schedule presets such as every-N-minutes or daily-at helpers.
- Provider/model override support through agent tools.
- Human approval workflow before creating enabled jobs.
- New job file format or migration.
- Changing scheduler API/CLI contracts.
- Changing how scheduled runs execute beyond existing session id behavior.
- Building a web UI for hot reload status or reload errors.
- Distributed/multi-process hot reload coordination.
- Filesystem event watcher implementation for hot reload.

## Further Notes

This feature should preserve the existing scheduler ownership model: each agent owns jobs under its workspace. Agent tools are another entrypoint into the same scheduler service, not a separate scheduler implementation.
